# Documentation & Showcase Infrastructure (DS-1E)

The audit, architecture decisions, and certification record behind [docs/DOCUMENTATION.md](../DOCUMENTATION.md) — read that document for the contributor-facing how-to; this one is the evidence and reasoning for whoever next has to decide whether to change it.

## 1. Documentation infrastructure audit

**Registry** (`src/lib/design-system-navigation.ts`, 369 lines before this phase): `NAV_SECTIONS` (6), `NAV_GROUPS` (28), `NAV_REGISTRY` (70 entries). `NavEntry` has 6 required fields (`id, title, href, section, group, order, description`) and 5 optional (`badge, previous, next, related, aliases`). Query helpers (`getEntry`, `getEntryByHref`, `getSection`, `getGroup`, `getGroupsForSection`, `getGroupEntries`, `getSectionEntries`, `getPrevious`, `getNext`, `getRelated`) are the documented single read path — "every docs component reads through these, never NAV_REGISTRY directly" (the file's own comment). True for id/section/group lookups; not true for `getRelated` (§3).

**Source-of-truth relationship**: two parallel systems, kept in sync entirely by hand. Nothing generates `page.tsx` files from the registry, and nothing generates registry entries from routes. Verified both directions: every registry `href` resolves to a real `page.tsx` (zero missing), and of 71 real `page.tsx` files, 70 are registered — the 71st (`src/app/compositions/frame/page.tsx`) is a deliberately unregistered iframe embed target, with its own doc-comment explaining why. So route/registry drift does not exist today — but nothing enforced that; it was true by discipline, not by construction. It is now enforced (`design-system-navigation.test.ts`, "route integrity" suite).

**`src/components/docs/` (10 components)**: `DocsShell`, `DocsSidebar`, `DocsSidebarGroup`, `DocsPageHeader`, `DocsPageNavigation`, `DocsLinkCard`, `DocsRelatedGrid`, `DocsGroupCard`, `DocsSearchTrigger`, `DocsTableOfContents`. One location smell worth recording: four more shared docs-shell primitives (`DocsSectionLanding`, `DocsLandingSummary`, `DocsEntryGrid`, `DocsShowcase`) live under `src/app/docs/_components/` — a Next.js private route folder scoped to `/docs` — despite being imported by 15+ pages across other route groups (`application-components`, `playground`, `foundations`, `applications`, `marketing-components`, `tokens`, `core-components`, `workflow-patterns`). Not moved this phase (a file relocation with 15+ import-site updates is exactly the kind of broad mechanical change this phase's brief scopes out — "do not redesign documentation"); recorded here as a real, low-risk future cleanup.

**Search**: already derives entirely from `NAV_REGISTRY` (`DocsSearchTrigger.tsx`) — no separate index, no duplicated definitions, which is the property Part 7 asked to confirm. But the match predicate (`CommandPalette.tsx`, pre-phase) matched only `item.label` (the title) — `description` was never searched, and `aliases` (present on 8 registry entries, with its own doc-comment describing it as "worth surfacing in search/wayfinding") had zero consumers anywhere in `src/`. Fixed (§4) by adding a generic `keywords` field to `CommandPaletteItem` and populating it with `[description, ...aliases]` in `DocsSearchTrigger` — `CommandPalette` itself stays domain-agnostic (doesn't need to know `NavEntry`'s field names).

**Page registration boilerplate**: every page follows `const entry = getEntry("id")!` at module scope, then `<DocsShell entry={entry}>` wrapping `<DocsPageHeader entry={entry}>`. The one, real, evidenced duplication: **54 `page.tsx` files** independently hand-build a related-links array as `[getEntry(id)!, getEntry(id2)!, ...]`, none of them via `entry.related`/`getRelated()`. Concretely: `foundation-components`'s registry entry declared `related: ["foundation-layout", "foundation-table"]` while its own page hand-typed `relatedComponents = [getEntry("foundation-layout")!, getEntry("foundation-table")!, getEntry("foundation-metadata")!]` — a third item the registry's own field never knew about. Two mechanisms, already drifted, on the very entry that happens to populate `related` at all. (`getRelated()` itself: defined, zero call sites anywhere in `src/` outside its own file.) The literal-object crash-risk pattern from the earlier DS-9.1 pass (`relatedComponents = [{title, href, description}, ...]` typed inline) is confirmed gone — every current instance goes through `getEntry(id)!` — but the residual risk moved, not disappeared: a typo'd id in `getEntry("typo")!` still compiles (the `!` is compile-time-only) and only crashes at runtime, on whichever page renders it. All 70 ids in use today check out against the registry; nothing enforced that either, until now (`design-system-navigation.test.ts`, "getEntry() reference integrity" suite).

Several section-landing pages (`application-components`, `playground`, `foundations`, `applications`, `marketing-components`, `core-components`, `docs` (architecture), `workflow-patterns`) also hand-author `purpose`/`whatYoullLearn`/`stats` props per page (`DocsSectionLanding.tsx`'s prop contract), including literal hardcoded counts in `stats` (e.g. `"Tiers documented": "4"`, `"Platform templates": "7"`) that comments describe as "grep-verifiable" but are not code-verified — a real, low-severity drift risk, not fixed this phase (fixing it would mean either deriving every stat programmatically, a larger change than warranted for descriptive copy, or building a generic stats-verification harness — recorded as a future recommendation, §6).

**Showcase/example registration** (`src/workflows/`, `src/platforms/`, `src/production/`, `src/capabilities/`): confirmed zero metadata layer. Each family's core type (`Workflow`, `PlatformArchitecture`, `ProductionPipeline`, `CapabilityRegistry`) has no `category`/`tags`/`difficulty`/`relatedPages`/`owningComponent` field; each `examples/index.ts` is a flat array of plain exported data objects consumed directly by whichever playground page imports them. No registry of "what examples exist." Built from scratch this phase (§4) — `src/lib/showcase-registry.ts`, demonstrated on one family only (`src/workflows/examples/showcase.ts`), not all four, per the brief's "do not migrate the whole site."

**Illustrations** (`src/illustrations/`): same absence, more minimal — `src/illustrations/index.ts` is a plain barrel with no catalog of which illustrations exist or which page owns them. Not built this phase; see §6.

**Existing validation**: none, confirmed. `scripts/verify.mjs` (DS-1D) has no docs-aware step; `packages/design-system/scripts/*.mjs` are scoped entirely to the publishable package's API surface; `test/` (DS-1C) had no nav/docs test file. `grep -rln "design-system-navigation" test/ scripts/ packages/design-system/scripts/` returned zero matches before this phase.

## 2. Page contracts

`src/lib/docs-contracts.ts`. The brief names seven archetypes (Landing, Reference, Pattern, Application, Architecture, Playground, Historical Reference); the registry's actual type system has only `NavPageType` (4 values) and `NavBadge` (8 values). Rather than adding a third, parallel classification field for a maintainer to keep in sync with the other two, `getArchetype(entry)` derives one of the seven names from the (pageType, badge) pair that already exists — verified against all 17 distinct (pageType, badge) combinations actually present across the 70 entries; every combination classifies unambiguously (`design-system-navigation.test.ts`'s "every registry entry classifies to exactly one archetype" test asserts this holds, not just that it held at audit time).

`PAGE_CONTRACTS[archetype]` specifies `expectedSections` (documentation of intent — a reviewer's checklist, not machine-checkable without JSX parsing), plus two fields the validator CAN check: `requiresBadge` and `requiresRelatedOrPrevNext`. Running the latter check against real data surfaced 7 entries with neither previous/next nor related — every one of them the sole entry in its own nav group (`foundation-layout`, `foundation-navigation`, `foundation-forms`, `foundation-overlays`, `filter-search`, `workflows-library`, `platform-architecture-doc`). A sole-entry group has no sibling to chain to by construction, and populating `related` for all 7 would mean adding a "related components" section to seven pages that don't render one today — page rewriting, out of scope. The check exempts sole-entry-group pages rather than silently passing or hard-failing on a structural non-issue; the exemption is derived from `getGroupEntries(entry.group).length === 1`, not a hand-maintained list, so a *future* sole-entry-group page is automatically exempted too, and a future multi-entry-group page that's actually missing grounding still fails loudly.

## 3. Showcase registration

`src/lib/showcase-registry.ts` — `ShowcaseMeta` (`id, title, category, tags, difficulty, relatedPages, owningComponent`) plus `defineShowcase()`, an identity function whose only job is being a greppable call-site convention. Deliberately a metadata layer *alongside* rendering data, not a change to `Workflow`/`PlatformArchitecture`/etc. themselves — those types already carry everything the illustration/motion engines need to render; adding showcase fields to them would conflate "how to draw it" with "how to catalog it." Piloted on `src/workflows/examples/showcase.ts` (the five pattern-gallery examples: `artwork-production`, `publishing`, `commerce`, `quality-assurance`, `multi-channel-publishing`) — one family, not all four, per brief.

## 4. Documentation validation

`src/lib/design-system-navigation.test.ts` — co-located with the registry it validates, per `docs/TESTING.md`'s own convention (`Button.tsx` → `Button.test.tsx`). This was a deliberate architecture choice over a new `scripts/check-docs.mjs`: the registry is a `.ts` module with real types, and Vitest already transpiles TS natively (unlike a plain `.mjs` Node script under CI's Node 20, which has no native TS support) — importing the real module and asserting on it is strictly more robust than regex-scanning its source text a second time. Being a normal Vitest file also means it needed **zero** new pipeline wiring: `vitest.config.ts`'s existing `include: ["src/**/*.{test,spec}.{ts,tsx}"]` picked it up automatically, so it's already exercised by `npm test` → `verify:fast` → `verify` → `verify:full` (DS-1D), satisfying Part 9's "integrate documentation verification into the verification pipeline" with no changes to `scripts/verify.mjs` or `.github/workflows/release.yml` at all.

19 tests across 5 suites: registry integrity (duplicate ids/hrefs, dangling previous/next/related/section/group references, duplicate NAV_SECTIONS/NAV_GROUPS ids), route integrity (registry↔filesystem in both directions, including a self-check that `INTENTIONALLY_UNREGISTERED_ROUTES` doesn't itself go stale), `getEntry()` reference integrity (scans all 71 `page.tsx` files' source text for `getEntry("...")` calls and checks each id against the real registry — the one check that reads page source rather than just registry data, because this is exactly the class of bug a `!` non-null assertion hides at compile time), page contracts (archetype classification completeness, badge/grounding requirements), and query-helper regression guards (`getPrevious`/`getNext`/`getRelated`/`getRelatedLinks` never throw, and `getRelatedLinks` returns a fully-resolved render-ready shape).

`getRelatedLinks(entry)` (added to `design-system-navigation.ts` alongside the pre-existing, previously-unused `getRelated()`) is the fix for the drift found in §1 — resolves `entry.related` directly into `DocsRelatedGrid`'s prop shape, so a page has no reason to hand-roll the mapping again.

## 5. Route integrity, coverage, search — see docs/DOCUMENTATION.md §6–8 for the user-facing summary

Route integrity: fully covered by `design-system-navigation.test.ts` (§4). Coverage: `scripts/docs-coverage.mjs`, advisory-only (always exits 0, not wired into `verify`), reports entry counts by archetype and by section plus component-family export-statement volume for context — deliberately not a precision "% of components documented" figure, since (per this phase's own research) computing that precisely would require enumerating every gallery's underlying data file individually, out of proportion to what "do not require 100%" asks for. Search: fixed per §1 — `CommandPaletteItem` gained a generic `keywords?: string[]` field (kept domain-agnostic; `CommandPalette.tsx` doesn't need to know about `NavEntry`), populated by `DocsSearchTrigger` with `[entry.description, ...(entry.aliases ?? [])]`, and the filter predicate now checks it alongside the title.

## 6. Known limitations

- **Content-shape validation doesn't exist.** `PAGE_CONTRACTS[archetype].expectedSections` documents what a page of that archetype should contain; nothing parses a page's rendered JSX to confirm it actually does. This is a real, acknowledged boundary — DS-1F (Component Certification) is the natural owner of that class of check, per the brief's own success criterion ("DS-1F can rely on documentation infrastructure rather than manually auditing pages" — this phase built the registry-level half of that reliance, not the content-level half).
- **`src/app/docs/_components/` location smell not fixed** (§1) — four widely-shared docs primitives sit in a route-private folder. Low risk, real cleanup, scoped out as a mechanical multi-file move unrelated to this phase's actual goal.
- **Showcase registration covers one example family** (`workflows`), not all four (`platforms`, `production`, `capabilities` still unregistered) — per brief, demonstrating the pattern rather than migrating everything.
- **Illustrations have no registry at all** — audited, not built. Building one would be a larger net-new system (the brief's "do not create large new documentation sections" cuts against inventing a whole new registry in the same phase that's supposed to be consolidating an existing one).
- **Hardcoded `stats` literals in section-landing pages** (e.g. `"Platform templates": "7"`) are not verified against real counts — found, not fixed; a generic stats-verification harness is a reasonable, separate future task.

## 7. Future recommendations

1. Move `src/app/docs/_components/{DocsSectionLanding,DocsLandingSummary,DocsEntryGrid,DocsShowcase}.tsx` into `src/components/docs/`, updating their 15+ import sites — a mechanical, low-risk cleanup once someone has a reason to touch those files anyway.
2. Extend `getRelatedLinks`/registry `related` adoption to the remaining 53 pages still hand-building `[getEntry(id)!, ...]` arrays — safe to do incrementally, file by file, with `design-system-navigation.test.ts` already catching any id mistakes made along the way.
3. Extend showcase registration (`src/lib/showcase-registry.ts`) to `src/platforms/examples/`, `src/production/examples/`, and `src/capabilities/examples/` — same pattern as `src/workflows/examples/showcase.ts`.
4. Consider a lightweight stats-verification test for the hardcoded literal counts in section-landing pages' `stats` props, once there's a clear derivation source for each (some are legitimately manual editorial claims, not counts of anything in the registry).
5. If illustration cataloging becomes a real pain point (not evidenced as one yet), design a registry for `src/illustrations/` following the same "metadata alongside rendering data" shape `showcase-registry.ts` established for examples.
