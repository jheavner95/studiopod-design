# 13. Token Consolidation & Release Readiness (DS-4)

DS-4's brief: audit every active token source, produce an ownership map, consolidate what's genuinely duplicated or dead, fix hardcoded values that should be tokens, verify the theme contract, verify package distribution, document the real architecture, and assess release readiness — without redesigning components, migrating application pages, or beginning DS-5.

## Part 1 — Ownership map

| Layer | File(s) | Owns | Mechanism |
|---|---|---|---|
| Primitive color | `src/styles/palette.css` | Raw 11-step ramps (`--palette-{slate,blue,green,amber,red}-{50..950}`, plus neutral/white/black) | Plain `:root`, deliberately **outside** `@theme` — no ramp-utility bypass of the semantic layer |
| Semantic color, radius, shadow, section spacing, containers, fonts, easing | `src/styles/theme.css` | Everything that should generate a Tailwind utility class | Inside `@theme { ... }` |
| Motion (raw scale) + z-index | `src/styles/tokens.css` | `--duration-*`, `--motion-{duration,delay,distance,ease}-*`, `--z-*` | Plain `:root`, deliberately outside `@theme` — consumed as raw custom properties by JS (framer-motion), not via generated utility classes |
| Fluid type scale | `src/styles/typography.css` | `.text-display-1` … `.text-metadata` | Plain classes (size/line-height/letter-spacing travel together, not exposed as separable `@theme` font-size tokens) |
| Composable non-utility classes | `src/styles/utilities.css` | `.focus-ring`, `.glass-panel`, `.bg-grid-dots`, `.bg-grid-lines`, `.scrollbar-none` | Plain classes |
| Import order + Next's bg/fg convention | `src/app/globals.css` | `@import "tailwindcss"` then palette → theme → tokens → typography → utilities, plus a 2-line `@theme inline` aliasing `--color-background`/`--color-foreground` | — |
| JS mirror (older scale) | `src/lib/tokens.ts` | `motionDuration`/`motionEase` (5/3 keys), `zIndex`, `accentRgb`/`successRgb` | Hand-maintained mirror of `tokens.css`'s `--duration-*`/`--ease-*` (theme.css) |
| JS mirror (newer scale) | `src/motion/tokens.ts` | `motionDuration`/`motionEase`/`motionDelay`/`motionDistance` | Hand-maintained mirror of `tokens.css`'s `--motion-*` scale — the vocabulary `src/motion`'s primitives actually use |

There is **no `tailwind.config.ts`** — pure Tailwind v4 CSS-first, config lives entirely in the files above.

## Part 1 — Duplication and ambiguity findings

**No CSS-variable duplication between app and package.** `packages/design-system/src/styles.css` is a literal `@import` of `src/styles/*.css` by relative path — one copy of every custom property in the repo, enforced by `packages/design-system/scripts/check-css.mjs` (asserts 8 canonical markers, one per source file, survive into `dist/styles.css`, because tsup's PostCSS integration can otherwise silently run the bundled CSS through Tailwind's compiler in isolation and strip the whole `@theme` block as "unused"). This was the first thing to rule out, and it's a non-finding — nothing to merge there.

**Real duplication found and addressed this phase:**
- `Grid.tsx` and `CardGrid.tsx` declared byte-identical `GridGap`/`gapMap` (`sm: gap-4, md: gap-6, lg: gap-8`). Consolidated — `CardGrid` now imports both from `Grid`, since it's already documented as a fixed-breakpoint specialization of it.
- `--color-ink-inverse` (`theme.css`) had zero usages anywhere in `src/` or `packages/design-system/src/`, and no documentation-page presence — removed as genuinely dead, undocumented surface.
- `max-w-[2160px]`, a raw magic number, was duplicated verbatim in `GlobalNav.tsx` and `DocsShell.tsx` with no comment explaining it (and it's *larger* than `--container-wide`, so it isn't a content-container value at all — it's an outer site-shell bound). Promoted to `--container-shell` in `theme.css`; both call sites now reference the token.
- The public tokens page (`SpacingSection.tsx`) documented only Tailwind's built-in numeric spacing scale, under the description "every gap, padding, and margin in the system is a multiple of it" — true, but silently omitting the project's *own* fluid spacing tokens (`--spacing-gutter`, `--spacing-section-{xs..xl}`), which `Container`, `GlobalNav`, `DocsShell`, and every `SectionShell` actually reach for. Added a second section documenting them.

**Real duplication found and deliberately left alone, with reasoning:**
- **Two parallel motion-duration/-ease scales** (`tokens.css`'s older `--duration-*`/`theme.css`'s `--ease-*`, mirrored as `motionDuration`/`motionEase` in `src/lib/tokens.ts`; and `tokens.css`'s newer `--motion-duration-*`/`--motion-ease-*`, mirrored as `motionDuration`/`motionEase` in `src/motion/tokens.ts`) — same export names, different shapes, in two different modules. This looks alarming out of context but is **not a live bug**: the two modules are never imported into the same file (TypeScript would force disambiguation at the import site if they were), and each serves a genuinely distinct, already-load-bearing purpose — `lib/tokens.ts`'s scale feeds `accentRgb`/`successRgb` box-shadow pulse interpolation in `motion/variants.ts`, while `motion/tokens.ts`'s scale is the actual vocabulary every `src/motion` primitive (`FadeIn`, `SlideUp`, `Stagger`, …) is built on. Renaming either is a **public API change** — both are re-exported at `@studiopod/design-system/tokens` and frozen in `api-baseline/tokens.json` — and "Do not create aliases without a clear semantic purpose" argues against inventing a rename-plus-compat-alias here for a collision that costs nothing in practice. Documented instead of changed.
- **Five independent `paddingMap`/`gapMap` families** across `Card`/`Surface`/`Panel`/`GlassPanel`/`SurfacePanel` (padding) and `Stack`/`Inline`/`Grid`+`CardGrid`/`ContentColumns` (gap), all keyed `sm`/`md`/`lg`, disagreeing on what those labels mean in px (e.g. `md` is `gap-4` in `Inline`, `gap-6` in `Stack`/`Grid`, `gap-12` in `ContentColumns`). This is real inconsistency, but resolving it means changing seven-plus components' actual rendered spacing — squarely "redesign components," which this phase is explicitly scoped away from, and the audit couldn't determine from static reading whether the differences are deliberate per-family tuning (a card's padding rhythm and a page-column gap plausibly *should* differ) or accidental drift. Left as a documented, named finding for a future dedicated component-spacing audit, not silently resolved either way.
- **~25+ independent tone→Tailwind-class `Record`s** (`Alert.tsx`'s `FEEDBACK_TONE_TEXT`/`FEEDBACK_TONE_BG`, `PulseStatus.tsx`'s `toneMap`, `ProgressBar`/`ProgressRing`'s `TONE_FILL`/`TONE_STROKE`, and a long tail of `MARKER_TONE`/`LABEL_TONE`/`STATE_COLOR` Records across `workflow/`, `operational/`, `metadata/`) — the underlying *tokens* aren't duplicated (they all resolve to the same `success`/`warning`/`error`/`accent`/`neutral` semantic colors), but the mapping logic from a tone/status enum to a token is reimplemented per file. This is the same phenomenon engineering note 03 already names at the type level (`StatusTone` vs `FieldMessageTone`). A shared tone→class resolver is a legitimate future consolidation, but touches enough files (and enough type-vocabulary decisions already flagged as unresolved elsewhere) that it's out of scope here — named as a DS-5+ candidate, not attempted.
- **`--motion-duration-*`/`--motion-delay-*`/`--motion-distance-*` CSS custom properties** appear to have zero internal consumers (the motion engine reads the JS mirror directly, in seconds, never `var(--motion-duration-*)`), but the file's own comment states they're reserved "for the rare case a plain CSS transition needs the same rhythm." Removing intentionally-reserved, explicitly-commented infrastructure on the strength of "no current internal caller" is a different, riskier claim than removing genuinely undocumented dead weight like `--color-ink-inverse` was — left in place.
- **The public `zIndex` JS export** (`src/lib/tokens.ts`, frozen in `api-baseline/tokens.json`) has zero internal import sites (every real z-index usage reaches for the raw `"var(--z-toast)"` string directly in an inline `style` prop) — but it's public, versioned API surface that external consumers of `@studiopod/design-system/tokens` may already depend on directly. Removing unused-internally-but-public surface without evidence external consumers don't need it is exactly the kind of breaking change Part 3 says to avoid; left in place.
- **`--ease-standard`/`--ease-in-out-soft`/`--ease-linear`'s generated Tailwind utilities are architecturally present but never used in their plain form** — every call site writes the verbose `duration-[var(--duration-fast)] ease-[var(--ease-standard)]` instead, because `--duration-*` was deliberately kept outside `@theme` (so pairing it with a plain `ease-standard` class would read as an inconsistent shorthand). Fixing this cleanly would mean either moving `--duration-*` into `@theme` (enabling a plain `duration-fast ease-standard` idiom repo-wide) or leaving both arbitrary-value — either way it's a 60+-file mechanical rewrite for a purely cosmetic/DX win, not a correctness or duplication fix. Named as a good candidate for a dedicated future cleanup pass, not attempted here — it would be disproportionate touch-surface for "avoid excessive abstraction."

## Part 1 — Naming inconsistencies (documented, not changed)

- "Spacing" means two unrelated things depending on which doc you read — Tailwind's implicit numeric scale (no project-defined `--space-*` tokens exist) vs. this project's own `--spacing-gutter`/`--spacing-section-*` fluid scale. Both are real and both are now documented (see `docs/TOKENS.md` and the updated tokens page).
- Color-scale step numbering isn't consistent between layers: `palette.css` uses the full canonical 11-step scale; `--color-accent-*` only exposes a curated subset (`300/400/500/600/700` + `soft`); no other semantic color family uses numeric steps at all. This is a legitimate curation choice, not a bug, but worth naming so a future semantic color addition doesn't invent a third convention.
- `--color-error` is the one semantic color that is *not* a verbatim palette alias — it's `--palette-red-500` lightened for AA contrast against `error-soft`, per its own comment. Every other semantic color is byte-identical to a specific palette step. The derivation methodology is disclosed but not generalized into a rule; `docs/TOKENS.md` now states it explicitly as the one documented exception.

Rather than duplicate it here, the canonical token hierarchy, ownership-by-layer table, naming conventions, "adding/deprecating a token," "implementation constant vs. token," and consumer install/import guidance all live in **`docs/TOKENS.md`** — the public-facing reference Part 7 asked for. This note is the audit-and-decisions record; that file is the one meant to stay current as the token system evolves.
