# StudioPOD Design System Engineering Notes

This directory preserves the genuinely useful engineering knowledge captured during the design system's internal certification exercise — architectural conclusions, reusable design principles, measurable findings, diagrams, and implementation lessons — after the certification/audit pages themselves were deleted as obsolete implementation (RM-5).

These notes are plain repo documentation, not part of the Next.js app (`src/app/`) — they render nowhere and are not linked from site navigation. They exist for engineers working in this codebase later who never saw the original certification pages.

15. [Tone System Consolidation (DS-5B)](./15-tone-consolidation.md) — resolves the tone/status/severity-map duplication DS-4 flagged: eight byte-identical unexported `StatusTone` declarations traced to Badge exporting no tone type at all (now fixed at the root via `src/lib/tone.ts`), the `ProgressBar`/`ProgressRing` fill/stroke consolidation that caught a real Tailwind-static-scanning bug before it shipped, and which of the ~15 workflow-tier lifecycle vocabularies were deliberately left alone as genuine domain-specific status, not tone duplication.
16. [Layout Primitive Consistency (DS-5C)](./16-layout-primitive-consistency.md) — the full-family audit across all 17 structural layout primitives (API shape, composition, docs, examples, implementation, testing): the 15 previously-unexported `Props` interfaces now exported to match `Workspace`/`SplitView`'s own precedent, the `ContentColumns`/`CardGrid`/`Container`/`SectionShell` documentation gap closed (including the honest finding that `ContentColumns` has zero real adoption anywhere in this codebase), the 6-of-17 primitives with no test coverage now covered, and the historical related-links drift between the three sibling "Layout" nav pages traced to a hand-rolled array and fixed at the root.
17. [Feedback & Status Component Consistency (DS-5D)](./17-feedback-family-consistency.md) — the full-family audit across all 20 feedback/status components: confirms the family's four tone vocabularies (`StatusTone`/`FeedbackTone`/`ProgressTone`/`FieldMessageTone`) are genuinely distinct, not drift; fixes `StatusIndicator`'s one real accessibility gap (zero ARIA of any kind, now wired to the shared `feedbackRole()` convention); closes test coverage for the 14 of 20 components that had none; and documents `LiveRegion`/`useAnnounce`'s actual (narrow, correct) role after finding it had no presence in its own home family's docs.
18. [Navigation Component Consistency (DS-5E)](./18-navigation-family-consistency.md) — the full-family audit across all 17 navigation components: confirms the roving-tabindex split (present on `Tabs`/`TreeNavigation`/`Menu`, absent on `SideNavigation`/`TopNavigation`/`NavigationRail`/`ContextNavigation`) is the correct WAI-ARIA widget-vs-landmark distinction, not drift; fixes `NavigationRail`'s one real gap (no way to mark an item current outside scroll-spy mode) and the `icon`/`badge` fields `TopNavigation`/`NavigationRail` silently dropped despite `NavigationItem` already supporting both; closes test coverage for 14 of 17 components that had none; and documents (without fixing, given real regression risk to live doc-site chrome) `GlobalNav`'s internal desktop/mobile inconsistency.
19. [Data Display Component Consistency (DS-5F)](./19-data-display-family-consistency.md) — the final component-family consistency phase, auditing the Table family (14) and Metadata family (16) plus `StatCard`: fixes the audit's own "single most important finding" — `MetadataRow`/`MetadataField` ran as a parallel, semantically unequal div-based system alongside `DescriptionList`'s real `<dl>`, now fixed to real `<dl>`/`<dt>`/`<dd>` with zero visual change; closes documentation gaps for `TableFooter`/`EmptyMetadata`/`LoadingMetadata` (the latter two had zero footprint anywhere on the docs site); closes test coverage for 22 of 31 components that had none. Closes out the DS-5 series (spacing → tone → layout → feedback → navigation → data display).

Certification ceremony: pass/fail scorecards restating conclusions captured here, certification-level ladders, stale roadmaps, "systems reviewed" tallies, and the ~103 files of `_components`/`_data` implementation these notes replace. None of that implementation is preserved — only the knowledge it produced.
